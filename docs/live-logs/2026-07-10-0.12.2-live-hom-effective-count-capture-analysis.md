### 2. HoM legality and cost are correct in the committed post-cast capture

The real-save 0.12.2 capture shows:

This agrees with the pinned base-game ability definition:


### 3. A previous manual live cast doubled army rows and aggregate territory/sec

The committed 0.12.1 live-cast observation records approximately doubled visible affected army rows and aggregate territory production after a player-triggered HoM cast.

The user-supplied follow-up states that another live cast again doubled army and territory/sec. This is mechanically consistent with base-game `compoundUnit` semantics: every cast adds the current bank multiplied by `val - 1`; for `val = 2`, the current target count is added again.

### 4. A post-cast snapshot does not make a second simulated HoM delta zero

Phase 1 starts each branch from the captured immutable snapshot and applies one requested action at `t=0`.

Therefore, when HoM is legal and the captured affected army has positive production:

```text
HoM territory/sec delta versus WAIT
= affected territory/sec in the captured pre-action snapshot
```

A prior manual cast changes the starting counts. It does not consume or neutralize the value of a new simulated HoM cast.

Consequently, the committed explanation that delta `0` was expected merely because the save was already mirrored is rejected.

### 5. Expansion ETA zero can be correct

Phase 1 clamps Expansion territory remaining to zero when the captured/projected territory already meets the next Expansion cost. In that state an ETA of `0` is correct and is not evidence of the HoM defect.

### 6. Reserve provenance and non-mutation are currently correct

Current runtime labels live reserve provenance as:

```text
live-config:postNexusEnergyReserveSeconds
```

The committed exact-SHA and real-save evidence report read-only/non-mutation success. The next patch must preserve those properties.

## Current code path audit

### Canonical identity resolver

Current path:

```text
getLaboratoryHouseOfMirrorsAbility(game)
  -> ability.effect[]
  -> compoundUnit effects
  -> canonical base ids
```

This layer now appears substantially correct.

### Live unit resolution

Current path:

```text
getLaboratoryResolvedUnit(game, unitId)
  -> getGameUnit(game, unitId)
  -> scenario alias scan fallback
```

The exact runtime object returned for a promoted/suffixed visible army row has not yet been proven against the live save.

### Count capture

Current `buildLaboratoryArmySnapshot()` effectively does:

```js
const override = getScenarioUnitCountOverride(unit);
const count = Number.isFinite(Number(override))
  ? decimalFrom(override)
  : decimalFrom(unit?.count?.() || 0);
```

This is unsafe for diagnosis because several distinct states collapse to zero:

- a real resolved zero count
- a wrong runtime unit object whose count is zero
- a missing or throwing count path hidden by fallback behavior elsewhere
- a promoted/suffixed identity mismatch

The next patch must not silently treat an ambiguous or mismatched live count as a verified zero.

### Base-game `count()` semantics

Pinned base-game source shows that `Unit.count()` is not merely raw saved count. It is computed through `_countInSecsFromNow()` and the producer-path model, then capped.

That means the statement "`unit.count()` is inherently the raw lower-tier count" is not established by source.

The likely remaining failure is one of:

1. Laboratory resolves the wrong runtime unit object for the canonical effect target.
2. Runtime has multiple identity surfaces for the visible empowered/suffixed row and the current lookup selects the zero one.
3. A cache/object lifecycle distinction exists between the effect target and the object resolved through `getGameUnit()`.
4. Scenario alias compatibility has leaked into live identity selection.
5. The live follow-up payload reflects a different local code state than pushed `main`.

The next implementation must instrument and prove which case applies before choosing a fix.

### HoM effect object is a high-value authority

Pinned base-game source constructs each effect with:

```text
effect.unit = game.unit(data.unittype)
```

For `compoundUnit`:

```text
bank = effect.unit.count()
output = bank * (val - 1)
onBuy -> effect.unit._addCount(output)
```

The actual `clonearmy.effect[].unit` object is therefore the first runtime object that should be compared with Laboratory's independently resolved object.

The next patch should prove:

```text
effect.unit === getGameUnit(game, canonicalBaseId)
```

or explicitly record why they differ.

### Production capture

Current production path uses:

```text
unit.eachProduction()?.territory
contribution = floor(count) * effective territory/sec per unit
```

The browser-agent follow-up reports that per-unit production is nonzero. This supports the conclusion that production lookup is not the primary remaining defect.

Production must nevertheless be parity-checked against:

```text
unit.totalProduction()?.territory
```

where available.

Do not use `totalProduction / eachProduction` as an unqualified primary count fallback. It can at best prove `floor(count)` and may hide fractional-count semantics relevant after doubling.

## Verifier blind spot

The current 0.12.2 browser verifier does not provide a clean acceptance proof for the live count resolver.

It:

1. captures a browser snapshot,
2. passes that snapshot into `buildMeaningfulLiveFixtureSnapshot()`,
3. replaces the captured army with four hand-built rows,
4. uses ids such as `unit stinger`,
5. calculates contributions with JavaScript `Number`,
6. then verifies that the Phase 1 simulation produces a positive HoM delta.

That proves the simulation can double a supplied positive army model. It does not prove that `captureLiveSnapshot()` produced the positive canonical army model.

The next verifier must make its legal-HoM acceptance source the unmodified result of the production capture path.

A synthetic engine-only fixture may remain as a separate unit test, but it must not satisfy the live-resolver acceptance gate.

## Required simulation contract

For a legal HoM action:

```text
countAfter = countBefore * 2
affectedAfter = sum(floor(countAfter) * perUnit)
totalAfter = unaffectedBefore + affectedAfter
deltaVsWait = totalAfter - totalBefore
```

When affected production before action is positive:

```text
deltaVsWait > 0
```

For exact integer counts without fractional edge cases:

```text
deltaVsWait == affectedTerritoryPerSecondBefore
```

A legal action with positive captured affected production must never pass with zero delta.

- Patch the real runtime surfaces before capture.

### Gate C: Phase 1 engine

- Feed only a snapshot that passed Gate A/B.
