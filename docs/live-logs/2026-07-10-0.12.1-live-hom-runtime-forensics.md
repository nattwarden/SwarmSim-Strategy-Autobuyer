# 0.12.1 Live House of Mirrors Runtime Forensics

Date: 2026-07-10

Status: ROOT CAUSE CONFIRMED IN REPOSITORY CODE; MAIN-WORLD LIVE COUNTS NOT DIRECTLY READ BY BROWSER AGENT

## Scope

Investigate why the 0.12.1 Laboratory Live Runner captured House of Mirrors affected units with ids such as `unit swarmling v`, included `unit territory`, and produced zero affected army counts on a real save.

## Browser-agent limitation

The browser agent's read-only evaluator ran in an isolated world where these main-world globals were unavailable:

- `window.angular`
- `window.kbcSwarmBot`
- `eval`
- `Function`

The agent therefore did not directly read real unit counts through the Angular game service. It did inspect:

- the live DOM
- loaded SwarmSim source behavior
- the local 0.12.1 repository
- the exported real-save Laboratory result

This limitation affects only direct count confirmation. It does not affect the repository-level root cause, which is explicit in the current implementation.

## Executive root cause

`buildLaboratoryArmySnapshot()` mixes two incompatible identity systems:

1. live SwarmSim runtime unit ids, such as `swarmling`
2. scenario-harness/UI canonical keys, such as `unit swarmling v`

The function currently:

- scans every runtime unit whose tab is `territory`
- includes the `territory` resource because it also belongs to that tab
- stores `getScenarioCanonicalUnitKey(unit)` as both `unitId` and `canonicalRuntimeUnitId`
- marks every scanned territory-tab unit as affected by House of Mirrors

Phase 1 later indexes House of Mirrors units using the eleven base runtime ids. The snapshot rows are therefore keyed differently from the action model.

## Current faulty path

Function:

`buildLaboratoryArmySnapshot(game, warnings, formulaStatuses)`

Current behavior in the 0.12.1 runtime source:

```js
const territoryUnits = (game?.unitlist?.() || [])
  .filter((unit) => getTabName(unit) === "territory");

for (const unit of territoryUnits) {
  const canonicalRuntimeUnitId =
    getScenarioCanonicalUnitKey(unit)
    || normalizeLabelKey(unit?.name || "territory-unit");

  // row stored under scenario/UI key
}
```

This causes:

- `territory` resource inclusion
- tier/display suffixes in ids
- live/action map key mismatch
- zero or unresolved counts in later canonical lookups

## Correct runtime lookup contract

For each affected base id:

```js
game.unit("swarmling")
game.unit("stinger")
game.unit("spider")
```

The Laboratory resolver should use:

```js
getLaboratoryResolvedUnit(game, unitId)
```

with `getGameUnit(game, unitId)` as its direct live path and existing scenario aliases/overrides as compatibility fallback.

Snapshot identity rule:

```text
unitId = canonical base runtime id
```

Scenario/UI metadata may be stored separately, for example:

```json
{
  "unitId": "swarmling",
  "canonicalRuntimeUnitId": "swarmling",
  "scenarioCanonicalKey": "unit swarmling v",
  "label": "swarmling"
}
```

The scenario key must never be used as the action-model map key.

## Production contract

For every resolved affected unit:

```text
count = runtime count or explicit scenario override
effective territory/sec per unit = unit.eachProduction()?.territory
contribution = floor(count) * effective territory/sec per unit
```

Existing fallback:

`getTerritoryPerSecondPerUnit(unit)`

Affected total:

```text
sum(contribution for exactly the eleven HoM units)
```

Unaffected total:

```text
runtime total territory/sec - affected total
```

Required parity:

```text
affected + unaffected == runtime territory/sec
```

within documented Decimal tolerance.

## `unit territory` inclusion

`territory` is a resource unit with `tab: "territory"`. It is not a House of Mirrors target.

It is included only because the current function scans the entire territory tab instead of reading `clonearmy.effect[]` or iterating the verified affected-id list.

Correct exclusion rule:

```text
Only ids present in the resolved HoM compoundUnit effect set are affected.
```

No generic territory-tab scan should remain in the Laboratory HoM path.

## Clone Larvae control result

Clone Larvae resolves correctly because its path uses canonical runtime identity and direct effect data:

- `getGameUpgrade(game, "clonelarvae")`
- `getLaboratoryCompoundEffect(cloneAbility, "larva")`
- `getCurrentResource(game, "larva")`
- `getCurrentResource(game, "cocoon")`
- `effect.output()` for runtime preview

The first real-save run showed exact equality between source-verified Clone output and runtime preview.

This provides a useful implementation pattern for House of Mirrors: canonical ability id, runtime effect inspection, canonical unit ids, and separate formula parity.

## Reserve provenance defect

Current snapshot code hardcodes:

```js
reserveSource: "scenario-harness"
```

The live reserve value actually comes from:

```js
config.postNexusEnergyReserveSeconds
getVelocity(game, "energy") * reserveSeconds
```

Required provenance labels:

- live capture: `live-config:postNexusEnergyReserveSeconds`
- deterministic scenario with override: `scenario-harness`
- deterministic capture without override: `config:postNexusEnergyReserveSeconds`

The numeric reserve calculation should not change.

## Observation-summary defect

`laboratoryCreateObservationSummary()` only emits metric text when Clone Larvae or House of Mirrors has status `ok`.

When WAIT is legal and both abilities are `invalid-action` because of insufficient Energy, no lines are produced and the function falls back to:

`The live snapshot is incomplete or unavailable for metric-specific observation.`

This is incorrect for a valid Energy-blocked snapshot.

Required distinction:

- valid WAIT
- Energy-blocked Clone Larvae
- Energy-blocked House of Mirrors
- incomplete snapshot
- incomplete formula
- formula mismatch

An Energy-blocked summary should report current Energy, action cost, deficit, and that no metric comparison was performed because the action was not legal at t=0.
