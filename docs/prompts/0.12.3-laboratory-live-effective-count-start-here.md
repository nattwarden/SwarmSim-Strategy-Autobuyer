## Phase 1: shared HoM target descriptors

Refactor the current affected-id resolver into, or supplement it with, a shared descriptor resolver that retains both canonical identity and the authoritative runtime effect target.

A descriptor should be able to represent:

```json
{
  "unitId": "stinger",
  "effectIndex": 1,
  "effectUnit": "runtime object reference",
  "runtimeUnitName": "stinger",
  "runtimeSuffix": "V",
  "resolutionSource": "clonearmy-effect-definition"
}
```

Rules:

1. Canonical `unitId` remains one of exactly eleven base ids.
2. `effect.unit` from `clonearmy.effect[]` is the first live runtime authority.
3. Compare it with independent `getGameUnit(game, unitId)` resolution.
4. Export mismatch diagnostics when the objects/counts disagree.
5. Scenario/UI aliases remain metadata or harness compatibility only.
6. Never use `unit ...` or a display label as the Phase 1 map key.
7. Runtime effect set parity must be set parity, not accidental order dependence.
8. `territory` is never an affected target.

## Phase 2: effective count resolver

Create one shared count-resolution function for HoM capture, for example:

```js

  game,
```

Return explicit diagnostics, for example:

```json

  "runtimeUnitName": "stinger",
  "runtimeSuffix": "V",
  "scenarioOverrideApplied": false,
  "identityParity": true,
  "countParity": true
}
```

Required behavior:

1. When scenario harness is active, an explicit scenario override may be authoritative.
2. In live mode, prefer the count from the actual `clonearmy` effect target once its identity is proven.
3. Compare the effect-target count with independent canonical lookup.
4. Compare `floor(count) * eachProduction.territory` with `totalProduction.territory` when available.
5. A real verified zero count is valid.
6. A wrong-object zero, unavailable count, identity disagreement, impossible production parity or ambiguous runtime target is not valid zero evidence.
7. In those cases export `count: null`, a clear resolution status, and mark formula status incomplete or mismatch.
8. Do not infer counts from formatted UI text.
9. Do not silently infer full count from `totalProduction / eachProduction`; that proves at most a floored production count and can hide fractional semantics.
10. Preserve Decimal arithmetic throughout.
