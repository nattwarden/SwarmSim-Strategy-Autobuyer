# Git Verification Protocol

Status: mandatory repository process for implementation and formal verification work.

This protocol exists so GitHub can function as the actual source of truth for every AI agent and every later chat. It separates implementation, verification, generated evidence, and final reporting into states that can be reconstructed from Git history.

## Scope

Use this protocol for:

- runtime or strategy changes
- versioned fixes
- verifier or scenario changes
- formal release verification
- any task that will end with a VERIFIED or equivalent acceptance claim

A trivial documentation-only edit does not require a separate evidence commit unless the task explicitly asks for formal verification.

## Definitions

### Implementation commit

The commit containing the completed code, tests, verifier logic, fixtures, version changes, release notes, and other authored inputs needed to run verification.

Generated evidence from the final verification run must not be mixed into this commit.

### Pure verification command

A command that evaluates the repository but is not intended to modify tracked files.

Examples may include syntax checks, guardrail checks, build checks, and deterministic checks that only print results.

### Evidence-generating command

A command that intentionally rewrites generated JSON, CSV, Markdown, screenshots, summaries, hashes, timestamps, or other tracked evidence artifacts.

A command is evidence-generating because of what it does, not because of its name. Never assume a command named `verify` is read-only. Inspect the script and the active work order first.

### Evidence allowlist

The exact paths that a verification run is expected to create or rewrite.

The allowlist must be known before the command runs, either from the verifier source, package script, or active work order.

### Evidence commit

A separate commit created only after the full verification suite has passed against the exact implementation commit. It may contain only the approved generated evidence and a provenance record.

## Mandatory sequence

### 1. Start from synchronized Git state

Run:

```bash
git fetch origin
git checkout main
git pull --ff-only origin main
git status --short
git rev-parse HEAD
git rev-parse origin/main
```

Before implementation begins:

- `HEAD` must equal `origin/main`
- the working tree must be clean
- the agent must not invent local branch, commit, or push status

### 2. Complete and commit the implementation

Finish the implementation and all authored test/verifier changes.

Before committing:

- build the canonical userscript when required
- include generated runtime source that is part of the product
- include verifier code and static fixtures needed to reproduce the tests
- do not include evidence generated by the final acceptance run

Create and push the implementation commit.

Record:

```text
IMPLEMENTATION_SHA=<full 40-character SHA>
IMPLEMENTATION_TREE_SHA=<full tree SHA>
```

Do not amend, rebase, force-push, or otherwise rewrite this commit after verification begins. A required correction creates a new implementation commit and restarts verification from the new SHA.

### 3. Re-synchronize before formal verification

Run:

```bash
git fetch origin
git checkout main
git pull --ff-only origin main
git rev-parse HEAD
git rev-parse origin/main
git rev-parse HEAD^{tree}
git status --short
```

Hard gate:

- `HEAD` equals `IMPLEMENTATION_SHA`
- `origin/main` equals `IMPLEMENTATION_SHA`
- the tree SHA equals `IMPLEMENTATION_TREE_SHA`
- `git status --short` is empty

If any gate fails, stop. Do not claim verification.

### 4. Record toolchain and provenance

Before the first verification command, record at minimum:

```bash
git rev-parse HEAD
git rev-parse HEAD^{tree}
node --version
npm --version
```

When a verifier supports a commit environment variable, pass the exact implementation SHA so generated evidence embeds it. For example, set `GIT_COMMIT` to `IMPLEMENTATION_SHA` using syntax appropriate to the current shell.

Evidence that says `Commit: unknown` is incomplete provenance and must not support final acceptance by itself.

### 5. Classify every command before running it

For each required command, declare one of:

```text
PURE CHECK
EVIDENCE GENERATOR
```

For an evidence generator, list its exact evidence allowlist before execution.

The current working tree is allowed to become dirty only in allowlisted evidence paths produced by a declared evidence generator.

### 6. Run commands separately and capture results

For every command, record:

- exact command
- exit code
- raw final summary
- `git rev-parse HEAD`
- `git status --short`

After a pure check:

- the tracked working tree must still be clean
- any tracked modification is a hard stop

After an evidence generator:

- tracked changes are allowed only in the predeclared evidence allowlist
- changes to runtime code, verifier code, fixtures, package files, configuration, release notes, or other non-allowlisted files are a hard stop
- a real product/build change means the implementation commit was incomplete; create a new implementation commit and restart the whole protocol

Never silently restore unexpected changes and continue. First report and classify them.

### 7. Dependency and temporary files

Dependency directories and transient test output must not pollute Git status.

At minimum, `node_modules/` must be ignored by Git. Add other transient tool output to `.gitignore` when it is not intended as committed evidence.

Do not use destructive cleanup commands against an unknown dirty tree.

### 8. Complete the entire suite before committing evidence

Do not create the evidence commit after only one passing command.

Run every command required by:

- `AGENTS.md`
- `AI.md`
- `package.json`
- the active work order
- the changed subsystem's versioned verifier

If an evidence-generating command rewrites the same allowlisted files multiple times, only the outputs from the final complete passing run may be committed.

### 9. Inspect the final verification diff

After the full suite passes, run:

```bash
git status --short
git diff --stat
git diff --name-only
git diff --check
```

Hard gate before evidence commit:

- every tracked change is an allowlisted evidence file
- no implementation, test, verifier, fixture, configuration, or package file changed
- no unexpected untracked files remain
- evidence identifies the exact implementation SHA and relevant hashes

### 10. Create a separate evidence commit

Commit only the approved evidence files and provenance record.

The evidence commit message should clearly identify that it contains exact-SHA verification evidence.

Push the evidence commit normally. Never force-push.

The resulting history must be reconstructable as:

```text
<implementation SHA>  completed implementation and verifier inputs
<evidence SHA>        evidence generated by tests run against implementation SHA
```

### 11. Final synchronization gate

After pushing evidence, run:

```bash
git fetch origin
git rev-parse HEAD
git rev-parse origin/main
git status --short
git show --stat --oneline HEAD
```

Final reporting is permitted only when:

- `HEAD` equals `origin/main`
- the working tree is clean
- the evidence commit is after the implementation commit
- the evidence commit changes only approved evidence/provenance files
- the evidence names the exact implementation SHA it verifies

## Reporting rule

Do not report a formal final verdict from:

- a dirty working tree
- an unpushed commit
- a test run performed before the implementation commit existed
- a test run whose exact implementation SHA is unknown
- evidence mixed into the implementation commit
- a partial suite

Intermediate status may be reported plainly, but it must be labeled as intermediate and must not use final acceptance language.

The final report must distinguish:

```text
Implementation SHA:
Evidence SHA:
HEAD:
origin/main:
Working tree:
Commands run:
Exit codes:
Evidence files:
```

## Expected evidence changes are not implementation changes

An evidence-generating verifier may legitimately update timestamps, normalized live drift values, snapshot hashes, experiment hashes, Markdown summaries, CSV exports, or Copy Summary examples.

Those changes do not invalidate the run when all of the following are true:

- the command was declared an evidence generator
- the paths were allowlisted before execution
- no implementation or verifier source changed
- the full suite passes
- the generated outputs are committed separately after the run
- the evidence records the exact implementation SHA

Unexpected changes outside the allowlist remain a hard stop.

## Core principle

Commit and push the product first. Verify that exact immutable commit. Then commit only the evidence produced by that verification. Report completion only after GitHub contains both commits and the local tree is synchronized and clean.
